Codama Boundary Demo

What Codama already replaces, and what remains outside

This is the smallest concrete walkthrough of the architecture we ended up with. Codama carries instruction execution truth. The indexing spec owns read semantics. The runtime is reduced to compute and write prep. The runner only composes steps linearly.

Open live runner flow Open runner registry

Live flow

  1. Resolve pair inputs like USDC -> SOL from indexed token and pool data.
  2. Run a deterministic quote compute.
  3. Draft the Orca swap from Codama-backed instruction metadata.
  4. Optionally submit in wallet from the Runner view.

Recommended demo in the app: Draft Swap on Orca.

Boundary

Codama Instruction-level source of truth accounts, signers, PDA/default accounts
Indexing spec Off-chain read and discovery contract pool search, feeds, market series
Runtime spec Deterministic compute and write prep quote math, draft construction
Runner Tiny linear orchestration index_view -> compute -> contract_write

Specs to inspect

Codama IDL /idl/orca_whirlpool.codama.json

Instruction truth: accounts, defaults, PDA metadata.

 Indexing Spec /idl/orca_whirlpool.indexing.json

Indexed reads, projections, and canonical search/feed semantics.

 Runtime Spec /idl/orca_whirlpool.runtime.json

Compute and contract-write capabilities only.

 Runner Spec /idl/draft_swap_orca.runner.json

Linear composition: token resolve, quote, then draft.
The interesting leftover after pushing work into Codama is no longer account resolution. It is mostly transaction-envelope logic and any flow that needs a fresh read between steps. That is why Codama instruction plans look like the natural next place to shrink the custom runtime even further.
